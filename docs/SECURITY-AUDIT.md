## 1. [HIGH] UI-driving tools (tap/type/swipe + ADB input_*) bypass the entire confirm-token and money hard-stop model
*agent-safety* — `brain/src/safety.ts:6-16, 74-93; brain/src/tools/tier1-accessibility.ts; brain/src/tools/tier2-shizuku.ts:31-33; app/src/main/java/dev/kris/clyde/bridge/LocalControlServer.kt:51-64`

**Problem:** tap, type_text, swipe, global_action, input_tap, input_text, input_key, uiautomator_dump, screencap are all in the SAFE set, so canUseTool (agent.ts:75) never requires a confirm() token for them. The money hardStop (safety.ts:79-82) is deliberately scoped to moneyTools = {shell, su_shell, open_url, share_text, delegate_to_gemini} and explicitly excludes UI driving — test/run.ts:43 asserts hardStop('tap', {nodeId:'Pay Kim'}) === null. App-side, /a11y/tap, /a11y/type, /a11y/swipe, /a11y/global (LocalControlServer.kt:51-59) have no token check at all, and Tier-2 input_tap/input_text run straight through rish. Consequence: the single most dangerous prompt-injection path — a malicious SMS/notification/web page steering Clyde to tap a 'Pay'/'Send'/'Confirm' button inside a banking, wallet, or shopping app, or to type into another app's compose field — executes with zero per-action confirmation and is exempt from every money hard stop. The confirm()/argsHash model only protects clean intents (send_sms, open_url, etc.); it provides no protection once the agent is screen-driving, which is exactly the path injected on-screen content is most likely to trigger.

**Fix:** Treat screen-driving as consequential when it targets sensitive surfaces. Options that compose: (a) gate tap/type/input_* with a coarse confirm when the foreground package is in a sensitive-app set (banking/payments/wallets/app stores) — the app already knows the foreground package via the a11y tree; (b) require a confirm token before typing into or tapping inside a non-Clyde package that matches payment/auth heuristics; (c) at minimum, extend the hardStop money regex to also fire on UI-driving tools when the tap target text / typed text matches pay/confirm-payment/checkout phrases, accepting the 'Pay Kim' false positive as a confirm prompt rather than a hard block. Do not leave the screen-driving tier as an unconfirmed bypass of the confirm model.

## 2. [HIGH] Confirm sheet shows only brain-authored prose; the user never sees the actual params the token authorizes (consequential-action spoofing)
*android* — `app/src/main/java/dev/kris/clyde/overlay/OverlayController.kt`

**Problem:** The whole confirm/token design hangs on the user seeing what they approve, but ConfirmPanel renders ONLY ui.confirmSummary and ui.confirmDetails — two free-text strings supplied by the brain in the /confirm call (LocalControlServer.kt lines 91-99 pass body.summary/details straight through). The real params (pendingParams) are hashed into the token (argsHash, line 215) and later enforced against the fired intent (validateIssuedToken, line 138-142), but they are NEVER displayed. So the string the human reads and the args the token authorizes are decoupled. A compromised or prompt-injected brain can show summary='Open your bank's site' / details='chase.com' while sending params {url:'evil.com'}; the token binds to evil.com, the user taps Approve, and openUrl fires evil.com. The same applies to start_call (number), send_sms (to/body), share_text (targetPackage), and add_calendar_event. This directly violates the project's hard rule 'always show full URLs before opening links' and the safety model's intent that the user approves the actual effect. Because the app explicitly distrusts the brain elsewhere (fail-closed SAFE_INTENTS allowlist, independent argsHash), trusting brain prose for the human-facing approval is the weak link.

**Fix:** Render the security-relevant params in ConfirmPanel from pendingParams, not from brain prose: for open_url show the full normalized URL, for start_call/send_sms show the dialed number/recipient+body, for share_text show targetPackage. Build the confirm UI from the action name + params (the same object that is hashed), so what the user sees is exactly what the token authorizes. Keep the brain summary only as a secondary caption, never as the sole basis for approval.

## 3. [HIGH] Onboarding never sets Clyde as the Digital assistant app — the whole premise is left unconfigured
*install-ux* — `app/src/main/java/dev/kris/clyde/MainActivity.kt`

**Problem:** Clyde's reason for existing is to replace Gemini as the system assistant via the ACTION_ASSIST gesture (AndroidManifest.xml:72-88). But the first-run flow is Login → BrainSetup → Verify → Setup → Home, and none of those screens ever deep-links the user to the Digital-assistant-app picker. GrantsView (SetupScreen.kt:137-151) asks for runtime permissions, overlay, and accessibility, but NOT 'set Clyde as your assistant'. The only place the picker is wired is the Home 'Switch to Gemini' toggle (HomeScreen.kt:167 → GeminiRouter.openAssistantPicker), which is labeled as switching AWAY to Gemini, not toward Clyde. A user who completes the entire wizard will long-press home / swipe the assist gesture and still get Gemini, with no indication of why Clyde isn't responding. docs/permissions.md:113 and docs/build-phases.md:16 both say this step is required, yet the UI never performs it. This is the single biggest onboarding dead-end.

**Fix:** Add a GrantRow (or a dedicated final step) in SetupScreen.GrantsView that deep-links to Settings.ACTION_VOICE_INPUT_SETTINGS via GeminiRouter.openAssistantPicker, with copy like 'Set Clyde as your assistant — so long-press home / power summons Clyde instead of Gemini.' Detect whether Clyde is the current assistant (RoleManager.isRoleHeld(ROLE_ASSISTANT) on API 29+, or Settings.Secure 'assistant'/'voice_interaction_service') so the row shows done/not-done like the other grants, and gate the 'All set' button on it.

## 4. [MEDIUM] CLYDE_KEY shared secret leaked into process argv (and shell history) when syncing to Termux
*secrets* — `app/src/main/java/dev/kris/clyde/home/HomeScreen.kt`

**Problem:** The Home screen 'Sync' button interpolates the raw CLYDE_KEY into a shell command string and fires it across an app boundary via Termux RUN_COMMAND (TermuxRunCommand.runInTermux -> putExtra("com.termux.RUN_COMMAND_ARGUMENTS", arrayOf("-lc", command)) in app/src/main/java/dev/kris/clyde/bridge/TermuxRunCommand.kt:42). The command is `... sed -i "s/^CLYDE_KEY=.*/CLYDE_KEY=$k/" ...` / `echo "CLYDE_KEY=$k"`. The full secret therefore appears: (a) as Intent extras crossing to a separate app (observable by anything that can read the broadcast/intent on older or instrumented devices), and (b) in the argv of the bash/sed processes Termux spawns, i.e. /proc/<pid>/cmdline, which is world-readable to the Termux app and to any process on a rooted device, and may be captured by Termux session logging. The shared loopback secret is what authenticates every brain<->app call (X-Clyde-Key), so its disclosure lets a co-resident process drive the phone-control API.

**Fix:** Don't pass the secret as a shell argument. Either (1) write the key from the app side directly into the brain's .env over the already-authenticated loopback channel (the brain can expose a key-set endpoint, or the bootstrap can fetch it via the X-Clyde-Key-gated LocalControlServer it already serves), or (2) pass it to Termux via stdin / a 0600 temp file path rather than embedding it in the command argv. At minimum, use `printf '%s' "$KEY"` read from an env var passed through RUN_COMMAND's environment extra rather than literal interpolation into argv.

## 5. [MEDIUM] CLYDE_DEV_NOAUTH=1 fully disables auth on the brain's agent endpoint over loopback
*server* — `brain/src/server.ts`

**Problem:** authorized() returns config.devNoAuth when no key is set (server.ts:27). With CLYDE_DEV_NOAUTH=1, assertServerSafe() is satisfied without a key (config.ts:122) and EVERY route — including POST /query, which drives the full device-control agent (accessibility taps, intents, rish/su tiers) — is served with NO authentication to anything that can reach 127.0.0.1:8765. On Android, that is any app with INTERNET permission. This is an intentional dev escape hatch, but it is a single env-var away from running a fully unauthenticated remote-control surface, and there is no runtime guard tying it to a debug build. If it is ever left set in a shipped/sideloaded config, any co-installed app silently gains full agent control.

**Fix:** Make the no-auth path impossible in production: gate config.devNoAuth on a development signal that cannot be present in a release (e.g. require NODE_ENV !== 'production' AND a loopback-only bind AND log a loud persistent warning each request), or remove it entirely and require a key always. At minimum, in assertServerSafe() refuse to start if devNoAuth is set while any release indicator is present, and never honor devNoAuth when BRAIN_HOST is non-loopback.

## 6. [MEDIUM] Self-permission hard stop is trivially evaded; only matches the exact package substring on pm_grant/grant_signature_perm
*agent-safety* — `brain/src/safety.ts:85-91`

**Problem:** The 'never grant the app/itself new permissions' hard stop only triggers for tool === 'pm_grant' || 'grant_signature_perm' AND pkg.includes('dev.kris.clyde'). The same effect is reachable through tools that are NOT covered: the model can run the identical grant via the generic shell tool ('pm grant dev.kris.clyde <perm>') or su_shell ('su_shell {cmd:"pm grant dev.kris.clyde ..."}'), neither of which is checked by this branch. Those tools are consequential (need a confirm token) and their describeAction shows the raw command, so a watching user could catch it — but the stated invariant 'Clyde will not grant new permissions to itself' is not actually enforced; it is only enforced for two specific tool names, and the CLAUDE.md hard stop ('never grant the app new permissions without explicit user request') is not structurally guaranteed.

**Fix:** Move the self-grant block into a tool-agnostic check: in hardStop, for shell/su_shell/inject_event, scan the command string for a 'pm grant'/'appops set'/'pm grant ... dev.kris.clyde' pattern and block self-targeted permission grants regardless of which tool issues them. Equivalently, centralize a 'is this a permission-grant to our own package' predicate and apply it across shell, su_shell, pm_grant, and grant_signature_perm.

## 7. [MEDIUM] Money hard stop is an easily-circumvented English-phrase regex on transaction/egress tools
*agent-safety* — `brain/src/safety.ts:78-84`

**Problem:** hardStop's money guard is /\b(confirm payment|send money|transfer (funds|money)|place (an )?order|pay now|wire transfer|make (a )?payment|buy now|checkout now)\b/ tested against JSON.stringify(args).toLowerCase(). It is purely lexical and English-only: a delegate_to_gemini prompt or open_url that effects a payment without those exact phrases (e.g. 'complete the purchase', 'send $200 to @bob', 'venmo mom', a checkout URL with no 'pay now' literal, or any non-English equivalent) passes the guard. Because this is the only programmatic money stop and the rest of the defense is the model honoring the system prompt, the hard stop gives a false sense of a deterministic backstop while only catching a small keyword set. (It also only covers shell/su_shell/open_url/share_text/delegate_to_gemini — combined with finding 1, UI-driven payments are entirely uncovered.)

**Fix:** Do not rely on phrase-matching as the money backstop. Prefer a structural rule: route all egress that can move money through a confirm() whose summary the user must read (already true), and treat the regex only as an extra heuristic, not the guarantee. Broaden the pattern to currency/amount signals ('$', '€', amount + send/transfer/pay verbs) and known payment-app package/URL hosts, and document that the real protection is user confirmation, not the regex.

## 8. [MEDIUM] Zip-slip: zip entry names extracted with no path-containment check (path traversal out of staging dir)
*runtime* — `app/src/main/java/dev/kris/clyde/runtime/EmbeddedRuntime.kt`

**Problem:** In ensureInstalled(), every zip entry is written via `File(staging, name)` (line 71 for files, line 69 for directories) where `name` is the raw, unvalidated `entry.name` from the ZipInputStream. There is no canonical-path containment check (no getCanonicalPath().startsWith(staging) guard anywhere in the file — confirmed by grep). A zip entry named `../../shared_prefs/x.xml`, `../databases/...`, or any `..`-laden path resolves OUTSIDE the staging directory because Java's File(parent, child) honors `..` segments. Since the extracted tree is later renamed over the live prefix and node is exec'd from it, a tampered bootstrap zip can drop or overwrite files anywhere the app uid can write (app data dir, SharedPreferences, the brain entry, libtermux-exec.so), i.e. arbitrary-file-write leading to code execution under the brain. The bootstrap is gitignored and built out-of-band (bootstrap/inject-brain.sh re-zips arbitrary `find .` output with `zip -r`), so 'we control the zip' is a build-time assumption, not an enforced invariant — a classic zip-slip the extractor itself must defend against.

**Fix:** Before writing each entry (and each directory), resolve and verify containment: `val out = File(staging, name); val canonicalStaging = staging.canonicalFile; if (!out.canonicalFile.path.startsWith(canonicalStaging.path + File.separator)) throw SecurityException("zip-slip: $name")`. Reject (don't silently skip) any entry whose canonical path escapes staging, so a malformed/tampered bootstrap fails loud rather than writing outside the prefix.

## 9. [MEDIUM] android.util.Log calls are not stripped in release and the embedded brain.log captures full CLI stdout/stderr
*hygiene* — `app/build.gradle.kts`

**Problem:** The release buildType enables minify/shrink but proguard-rules.pro (only 4 -keep lines) adds no -assumenosideeffects for android.util.Log, and proguard-android-optimize.txt does not strip Log by default. So every Log.i/w/e (paths, 'brain started', accessibility/overlay failures, 'Speech recognition error', 'brain error') ships in the production sideload APK and is readable via logcat by any app holding READ_LOGS / by adb. Separately, BrainRunner redirects the embedded Node process's combined stdout+stderr (redirectErrorStream(true)) to filesDir/brain.log (BrainRunner.kt:59-60); that file accumulates everything the bundled claude-code CLI and Agent SDK print — which for an assistant that reads SMS/contacts/screen content can include sensitive material — and it is only truncated at 1 MB and never redacted.

**Fix:** Add `-assumenosideeffects class android.util.Log { public static *** d(...); public static *** v(...); public static *** i(...); }` (and w/e if desired) to proguard-rules.pro so debug logging is removed from release, and gate the verbose brain.log: only enable Node stdout redirection in debug builds (BuildConfig.DEBUG), or write to a size-capped, app-private file that is wiped on a clean shutdown rather than appended across restarts.

## 10. [MEDIUM] Embedded 'Sign in to Claude' button silently no-ops while the runtime is still unpacking
*install-ux* — `app/src/main/java/dev/kris/clyde/setup/BrainSetupScreen.kt`

**Problem:** In EmbeddedBrainSetup the Sign-in button's click handler is wrapped in `if (runtimeReady && !loginActive) { ... }` with no else branch. ## 11. [MEDIUM] Embedded brain setup has no timeout or failure path — a failed unpack/login spins indefinitely
*install-ux* — `app/src/main/java/dev/kris/clyde/setup/BrainSetupScreen.kt`

**Problem:** EmbeddedBrainSetup's LaunchedEffect polls `EmbeddedRuntime.isInstalled`, `BrainClient.healthz`, and `auth.isSignedIn` every 2s in an unbounded `while (true)` loop with no timeout and no error surface. If extraction fails (low storage, corrupt asset), if node can't exec (W^X / LD_PRELOAD failure — a known fragile assumption per CLAUDE.md), or if the brain crash-loops (BrainRunner.kt:39-49 just restarts with backoff), all three step cards stay on their 'in progress' text forever and the Continue button (line 282, gated on `online == true && signedIn`) never enables. The user is stranded with no diagnostic and no 'something went wrong' message. Contrast this with the Termux path's VerifyScreen, which at least has a 40-attempt timeout and a 'Check again' affordance (VerifyScreen.kt:54-61,128-137).

**Fix:** Mirror the VerifyScreen pattern in EmbeddedBrainSetup: bound the poll, and after a timeout show an error block ('Couldn't start the brain — check storage space / reinstall') with a retry and a way to view brain.log (BrainRunner writes filesDir/brain.log). Plumb BrainRunner/EmbeddedRuntime failures (already logged) into visible UI state rather than only Logcat.

## 12. [MEDIUM] Termux 'Open Termux' button fires RUN_COMMAND before allow-external-apps is enabled — fails silently on fresh installs
*install-ux* — `app/src/main/java/dev/kris/clyde/setup/BrainSetupScreen.kt`

**Problem:** In TermuxCompanionSetup step 2, the 'Open Termux' button calls TermuxRunCommand.runInTermux(ctx, 'echo paste the Clyde command…', background=false), which sends a com.termux.RUN_COMMAND intent. But Termux only honors RUN_COMMAND from external apps once `allow-external-apps=true` is in ~/.termux/termux.properties — and that line is written by the bootstrap script (clyde-bootstrap.sh:22-24) which hasn't run yet at this point. So on a freshly installed Termux, this button does nothing the user can see: Termux ignores the intent (or pops its own RUN_COMMAND permission/error), and runInTermux returns true regardless because startForegroundService succeeded (TermuxRunCommand.kt:46-53). The user taps 'Open Termux', the foreground app may not even change, and there's no feedback. The intended UX (open Termux so they can paste) is unreliable as the very first interaction.

**Fix:** Don't rely on RUN_COMMAND to open Termux before bootstrap. Launch Termux via a normal package launch intent (packageManager.getLaunchIntentForPackage('com.termux')) for the 'Open Termux' button, and reserve RUN_COMMAND for after the bootstrap has set allow-external-apps. Alternatively, drop the button and just tell the user to switch to Termux and paste — the clipboard already holds the command.

## 15. [LOW] CLYDE_KEY stored in plaintext SharedPreferences (no Keystore/EncryptedSharedPreferences)
*secrets* — `app/src/main/java/dev/kris/clyde/util/Prefs.kt`

**Problem:** The loopback shared secret is generated with SecureRandom (good) but persisted as a plain string in a MODE_PRIVATE SharedPreferences file (clyde_prefs.xml). It is not wrapped in the Android Keystore or EncryptedSharedPreferences, so it sits in cleartext at /data/data/dev.kris.clyde/shared_prefs/clyde_prefs.xml. App-private storage and the correct backup exclusions (allowBackup=false plus backup_rules.xml / data_extraction_rules.xml both excluding clyde_prefs.xml) mean this is not reachable on a non-rooted device or via adb backup, so the exposure is limited to a rooted/compromised device. Still, for a secret that authorizes full device-control, at-rest encryption is the expected hardening.

**Fix:** Store the key via EncryptedSharedPreferences (androidx.security.crypto) or derive/wrap it with an Android Keystore-backed key so it is not readable as cleartext from the prefs file on a rooted device. Keep the existing backup exclusions.

## 16. [LOW] Subscription enforcement and secret-logging hygiene verified correct (no leak)
*secrets* — `brain/src/config.ts`

**Problem:** Affirmative (non-)finding to avoid a false alarm: the brain robustly refuses off-subscription operation. assertSubscriptionAuth() fail-loud exits on ANTHROPIC_API_KEY, ANTHROPIC_AUTH_TOKEN, ANTHROPIC_IDENTITY_TOKEN(_FILE), ANTHROPIC_SERVICE_ACCOUNT_ID, on Bedrock/Vertex/Foundry/Mantle/AWS route flags, and on ANTHROPIC_BASE_URL; CLAUDE_CODE_OAUTH_TOKEN is correctly allowed as the subscription token. Separately, no secret is written to logs: the brain's console.error/console.log calls (config.ts, agent.ts:117, server.ts) print only variable NAMES and generic errors, never the CLYDE_KEY or token; the app never Log.* the key or credentials (grep clean); brain.log captures only the brain's non-secret stdout/stderr; only brain/.env.example (a placeholder) is git-tracked. ## 17. [LOW] App control server (8766) has no request body-size limit (NanoHTTPD parseBody reads full body)
*server* — `app/src/main/java/dev/kris/clyde/bridge/LocalControlServer.kt`

**Problem:** Unlike the brain server, which caps requests at MAX_BODY = 256 KiB (brain/src/server.ts:15,38-42), the app's LocalControlServer applies no size limit before calling session.parseBody(map) (line 154). NanoHTTPD 2.3.1 reads the entire Content-Length (or chunked) body into a temp file / memory with no app-imposed cap. Because every loopback port on Android is reachable by any installed app holding INTERNET permission, and the X-Clyde-Key is the ONLY barrier, a caller that has obtained the key (or a compromised/buggy brain) can POST an arbitrarily large body to e.g. /a11y/type or /intent/* and exhaust memory/disk, taking down the foreground AgentOrchestratorService that owns the brain session and the overlay. Auth runs before parseBody (line 40 vs 46), so unauthenticated callers can't trigger it — which is why this is a resource-exhaustion / defense-in-depth gap rather than a pre-auth DoS.

**Fix:** Before calling session.parseBody, reject oversized requests: read session.headers["content-length"], and if it exceeds a small cap (e.g. 256 KiB, matching the brain) return a 413 Response.Status.PAYLOAD_TOO_LARGE without parsing. For chunked/unknown-length requests, either reject them or wrap parseBody so the accumulated size is bounded. Mirror the brain's MAX_BODY constant so both sides enforce the same ceiling.

## 18. [LOW] Loopback shared-secret is a static bearer token with no replay/nonce protection
*server* — `app/src/main/java/dev/kris/clyde/bridge/LocalControlServer.kt`

**Problem:** Both servers authenticate with the same constant 128-bit X-Clyde-Key sent verbatim in a header on every request (Prefs generates 16 random bytes via SecureRandom — adequate entropy). There is no per-request nonce, timestamp, or sequence binding, so any captured request is infinitely replayable. On pure loopback between processes this is largely mitigated (Android does not let an unprivileged co-installed app sniff another process's loopback traffic), and the consequential-action tokens are separately single-use/TTL/args-bound (OverlayController.kt:113-147), so the practical exposure is limited. It is still worth noting that the entire trust model rests on (a) loopback isolation and (b) the key never leaking — and the key IS surfaced to the user UI and copied to clipboard / written into brain/.env (HomeScreen.kt:122-142, BrainSetupScreen.kt:74), widening where it can leak.

**Fix:** Accept the loopback-bearer model but tighten leak surface: avoid placing the full key on the clipboard, prefer a one-shot in-memory handoff to the brain over a user copy/paste where possible, and consider rotating the key on each app cold start (regenerate in Prefs.init and re-push to the brain) so a leaked key has a bounded lifetime. If a stronger guarantee is wanted, add an HMAC-over-(method+path+body+timestamp) header with a small acceptance window to defeat replay.

## 19. [LOW] Brain constant-time key compare is preceded by a length check that leaks key length
*server* — `brain/src/server.ts`

**Problem:** safeEqual() returns false immediately when ab.length !== bb.length before calling timingSafeEqual. This leaks the secret's byte length via response timing. The key is a fixed 32-hex-char (16-byte) value so the length is effectively already known, and the attacker would need loopback access plus an oracle — so impact is minimal. The app side (LocalControlServer.kt:123, MessageDigest.isEqual) has the same property. Noting for completeness; the comparison is otherwise correctly constant-time.

**Fix:** Optional hardening: hash both the provided header and the stored key to a fixed-length digest (e.g. SHA-256) and timingSafeEqual the digests, so no branch depends on the attacker-controlled input length. Given the fixed, publicly-known key length this is low priority.

## 20. [LOW] System prompt documents the wrong confirm() signature, weakening the approval contract it teaches the model
*agent-safety* — `brain/system-prompt.md:33-36; brain/src/tools/meta.ts:18-24`

**Problem:** The system prompt instructs: 'call confirm({summary, details}). It shows the user an approval sheet and returns a one-time token.' But the real confirm tool schema (meta.ts) is confirm({action, params}) — there is no summary/details parameter; the summary/details are derived server-side by describeAction from action+params. The args-binding security property depends entirely on the model passing the exact action name and exact params to confirm() so registerToken binds argsHash(params). A model that follows the prompt literally and supplies {summary, details} would register a token bound to action=undefined and argsHash({}), which then fails every consequential call (rec.tool !== tool). In practice the tool's own description corrects this, but shipping a system prompt that contradicts the security-critical tool contract is fragile and undermines the 'summary must be specific and in the user's terms' guidance, since summaries are actually generated from describeAction, not authored by the model.

**Fix:** Update system-prompt.md to describe the actual contract: 'call confirm({action: "<the exact tool you will run>", params: {<exact args, no token>}}); Clyde renders the approval sheet from those and returns a one-time token bound to that exact action+args — pass the SAME params plus the token to the action tool.' Remove the implication that the model writes the summary/details.

## 21. [LOW] Accessibility service subscribes to typeAllMask events it never consumes (over-broad scope, all-app event firehose)
*android* — `app/src/main/res/xml/accessibility_service_config.xml`

**Problem:** accessibility_service_config.xml declares android:accessibilityEventTypes="typeAllMask", which makes the framework deliver every accessibility event (text changes, view focus, notifications, etc.) from every app on the device to PhoneControlAccessibilityService. But that service's onAccessibilityEvent(event) is an empty no-op (PhoneControlAccessibilityService.kt line 49) — it acts purely on-demand (dumpTree/tap/screenshot) and never reads events. Subscribing to typeAllMask therefore grants and exercises the broadest possible passive monitoring surface (a known data-exfiltration/keystroke-observation vector for a11y services) for zero functional benefit, enlarges what the service can passively observe, and wastes battery/IPC. It also worsens the privacy story the user is asked to consent to when enabling the service.

**Fix:** Narrow accessibilityEventTypes to only what is needed. Since onAccessibilityEvent is empty and the service is demand-driven, set it to a minimal type such as typeWindowStateChanged (or remove passive monitoring entirely if the on-demand rootInActiveWindow path suffices). Also drop flags the service does not use (flagReportViewIds is unused by dumpTree) to keep the declared scope tight.

## 22. [LOW] open_url normalizes/opens model-supplied URLs as an implicit VIEW intent with no scheme allowlist
*android* — `app/src/main/java/dev/kris/clyde/intents/DeviceIntents.kt`

**Problem:** openUrl() takes a brain-supplied url string, and if it does not start with 'http' it prepends 'https://', then fires an implicit ACTION_VIEW. The 'startsWith("http")' check is loose: a value like 'httpx://...' passes through unmodified, and there is no allowlist of schemes — a url such as 'intent://...#Intent;...;end', 'content://', or a custom app scheme is forwarded verbatim to ACTION_VIEW, letting the model launch arbitrary deep links / intent: URIs in whatever app claims them, not just web pages. Combined with the confirm-sheet not displaying the real URL, the user cannot see this. Impact is bounded (still token-gated) so this is low, but the input handling is too permissive for a 'show the full URL' product rule.

**Fix:** Parse the URL and allow only http/https schemes (reject or refuse to auto-prefix anything else); explicitly block intent:, content:, file:, javascript: and unknown schemes. Validate with Uri.parse(...).scheme rather than a startsWith("http") string test, and pass the validated absolute URL to the confirm UI so the user sees the real destination.

## 23. [LOW] Symlink and chmod failures are swallowed by runCatching with no logging (silent partial-extraction)
*runtime* — `app/src/main/java/dev/kris/clyde/runtime/EmbeddedRuntime.kt`

**Problem:** `runCatching { Os.chmod(out.absolutePath, 448) }` (line 76) and `runCatching { link.delete(); Os.symlink(...) }` (line 89) discard their exceptions entirely — no Log, no rethrow. If chmod 0700 on `bin/node` (or any executable) fails, the runtime is still committed (staging→prefix rename proceeds) and node may be non-executable or, conversely, left with broader perms than intended; if a symlink critical to node's module resolution fails, the brain silently breaks. Because the surrounding extraction only fails-loud on the final rename (line 93) and the asset open, a half-applied prefix can be renamed into place and marked installed (version file written), so the next launch sees isInstalled()==true and never re-extracts. This violates the project's stated 'fail loud' principle for the security-critical exec setup.

**Fix:** Collect failures from chmod/symlink and abort the whole extraction (throw out of the runCatching block so ensureInstalled returns false and no version file is written), or at minimum Log.e each failure. Critically, do not mark the runtime installed unless every executable was chmod'd and every required symlink created.

## 24. [LOW] No integrity verification of the bundled bootstrap asset before extraction/exec
*runtime* — `app/src/main/java/dev/kris/clyde/runtime/EmbeddedRuntime.kt`

**Problem:** isBundled() only checks the asset opens, and ensureInstalled() extracts and then exec's node from it with zero integrity check — no embedded SHA-256/expected-digest comparison against the asset bytes. The APK signature is the real trust anchor (assets inside a signed APK aren't independently mutable on a non-rooted device), so this is low severity, but combined with the zip-slip/symlink findings it means there is no defense-in-depth: a swapped asset (rooted device, repackaged APK, or a build-pipeline mistake — the zip is gitignored and assembled by inject-brain.sh) is extracted and run with LD_PRELOAD as-is. There is also no pinning that the extracted node/CLI match the documented claude-code 2.1.112 / Node 20 baseline.

**Fix:** Bake an expected SHA-256 of bootstrap-aarch64.zip into a build constant and verify the asset stream's digest in ensureInstalled() before extraction; refuse to extract on mismatch. This is cheap defense-in-depth around the W^X exec path and makes accidental/malicious asset swaps fail loud.

## 25. [LOW] voice.speak() and the brain-query callback run on the IO thread, not the main thread VoiceIO requires
*hygiene* — `app/src/main/java/dev/kris/clyde/service/AgentOrchestratorService.kt`

**Problem:** handle() launches on the service scope (Dispatchers.Main), but BrainClient.query internally wraps its whole body in withContext(Dispatchers.IO) (BrainClient.kt:48), so the onEvent lambda — including the 'final'/'error' branches that call voice.speak(it) — executes on an IO worker thread. VoiceIO's own KDoc says 'Create/call from the main thread.' (VoiceIO.kt:13). Android TextToSpeech is created on the main thread in VoiceIO.init but then has speak() invoked off-thread here. While TTS.speak is often tolerant, this is an unsynchronized cross-thread call to a UI/audio component and is exactly the class of intermittent crash/ANR that only shows up on-device. The overlay.* calls are safe because OverlayController marshals via onMain {}, but voice.speak has no such guard.

**Fix:** Marshal the speak call back to the main thread, e.g. wrap the 'final'/'error' handlers' voice.speak(...) in a main-dispatched block (scope.launch(Dispatchers.Main){...} or a Handler(Looper.getMainLooper()).post), or have VoiceIO.speak internally post to its own main handler the way OverlayController.onMain does.

## 26. [LOW] screenshotBase64 spins up a new single-thread Executor on every call and blocks a server worker up to 8s
*hygiene* — `app/src/main/java/dev/kris/clyde/service/PhoneControlAccessibilityService.kt`

**Problem:** Each /a11y/screenshot request calls Executors.newSingleThreadExecutor() for the TakeScreenshotCallback and then latch.await(8, SECONDS) on the NanoHTTPD worker thread. The executor is shut down after await, but if takeScreenshot never invokes either callback (driver hang) the latch times out and the executor is still shut down — acceptable — yet allocating a fresh thread-backed executor per screenshot is wasteful churn, and the 8s synchronous block ties up the calling HTTP worker. Under the brain's multi-step UI automation (repeated screenshots) this is a steady stream of thread creation/teardown.

**Fix:** Use a single reusable executor (or the service's main executor / a shared cached pool) instead of newSingleThreadExecutor() per call, and consider returning a structured timeout error instead of silently returning null when latch.await times out so the brain can distinguish 'no screenshot' from 'failed'.

## 27. [LOW] Embedded brain entirely lacks a startup self-test before being marked ready; ProcessBuilder failures only surface in logs
*hygiene* — `app/src/main/java/dev/kris/clyde/runtime/BrainRunner.kt`

**Problem:** maybeStartEmbeddedBrain() (AgentOrchestratorService.kt:58-65) calls brain.start() but nothing verifies the brain actually came up (e.g. BrainClient.healthz()). If launchOnce throws (W^X/LD_PRELOAD failure, missing node) it returns false and the supervisor loop breaks silently (BrainRunner.kt:42 `if (!launchOnce(...)) break`) — the supervisor exits, no restart, no user-visible error, and the only signal is a Log.e. The first the user learns of a dead brain is a generic 'brain unreachable' at query time. Also, a brain that starts but then exit-loops fast will hot-loop launchOnce with exponential backoff up to 30s but forever, never giving up or surfacing a permanent-failure state.

**Fix:** After start(), probe BrainClient.healthz() (with a short retry budget) and reflect a clear 'brain failed to start' state to the UI/notification instead of relying on logcat. Treat a hard launch failure (exception) distinctly from a clean exit, and cap restart attempts (e.g. after N rapid failures show an error rather than looping indefinitely).
